# 通用翻譯標準（臺灣正體中文）

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

## 1. 空間與排版 (Spacing)

* **中西文間距（盤古間距）：**全形中文與半形英數、變數（範例：`{user}`、`%s`）之間，須加上**一個半形空格**。
    * 正確範例：`距離活動開始還有 5 分鐘`。
    * 錯誤範例：`距離活動開始還有5分鐘`。

* **標點符號：**全形標點符號前後**不加**空格。
* **單位連接：**
    * **非英文字元單位**（範例：`%`、`°`）：數字與單位符號之間**不空格**。範例：`完成度 100%`、`角度 90°`。
    * **英文字元單位**（範例：`kg`、`m`）：數字與單位符號之間**需空格**。範例：`重量 10 kg`。
    * **例外：**若原文數字與單位間無空格，則**遵從原文格式**。

## 2. 標點符號規範 (Punctuation)

為了清晰可讀，請遵守以下規範：

* **刪節號「……」：**使用 `U+2026`。優先使用**兩個全形**（……），若有排版需求可使用**單個全形**（…）。重點是**保持專案內部一致**。前後不加空格。範例：`載入中……`、`載入中…`。
* **引號「 」：**優先使用單引號「 」，若內層還有引號則使用雙引號『 』。**請勿使用雙引號 “” 或單引號 ‘’**。
* **破折號「——」：**推薦使用佔兩個漢字寬度的 `U+2E3A`（⸺），但通常也使用兩個連續的 `U+2014`（——）來實現。呈現上應為一條在水平和垂直方向均位於字面正中的直線。若有排版需求可使用**單個全形** `U+2014`（—）。重點是**保持專案內部一致**。**嚴禁**以全形逗號或連字號（-）代替。
* **夾注號（）：**
    * 內容含中文時使用**全形（）**。範例：`已達成（共 5 級）`。
    * 僅包含純半形數字或代碼時使用**半形 ( )**。範例：`Level (10)`。
* **斜線號「/」：**優先使用**全形／**，連接片語可加空格（範例：`Windows 11 / macOS Sonoma`）。
* **冒號「：」：**使用**全形：**。通常用於標籤，範例：`名稱：`。
* **乘號「×」：**描述解析度或倍率時，應將英文字母 x 改為乘號。範例：`16×`、`1920×1080`。

## 3. 專有名詞與用語 (Terminology)

* **名詞翻譯原則：**
    * 若無標準譯名，優先保留原文，或採用「暫譯名 (Original Name)」格式。
    * 技術名詞（範例：`API`、`Cookie`、`UI`）若在社群中已習慣使用原文，可不必強行翻譯。

* **一致性原則 (Consistency)：**
    * 翻譯相關內容應優先從原始專案找譯文（範例：Prism Launcher 找 Minecraft、Morphe 找 YouTube 等）。

* **驗證原則 (Verification)：**
    * 翻譯需經過實機驗證（LQA），確保文字在介面中的顯示效果與長度適當，並符合當下語境。

* **在地化優先 (Localization First)：**嚴格禁止非臺灣地區習慣用語。
    * 常用詞彙對照：`最佳化`（非 `優化`）、`軟體`（非 `軟件`）、`螢幕`（非 `屏幕`）、`連線`（非 `連接`）、`選單`（非 `菜單`）。

* **單位與品牌 (Units & Brands)：**
    * **單位準確性：** 注意大小寫與定義差異。範例：`KB` vs `kB`、`GiB` vs `GB`、`Mbps` vs `MBps`。
    * **品牌名稱：** 嚴格遵守官方大小寫。範例：`macOS`、`iPhone`、`GitHub`。

* **易混淆詞彙 (Commonly Confused Terms)：**
    * **記錄 vs 紀錄：** 動詞用「記錄」（範例：`記錄會議內容`），名詞用「紀錄」（範例：`世界紀錄`、`會議紀錄`）。
    * **其他 vs 其它：** 一律使用「其他」。
    * **副本 vs 複本：** 相對於正本（範例：信件副本 `CC`）用「副本」，單純的檔案複製用「複本」（`Copy`）。
    * **回覆 vs 回復：** 回答用「回覆」（`Reply`），還原狀態用「回復」（`Restore`）。
    * **Support (支持 vs 支援)：** 依語境區分。立場、情感或贊助用「支持」（範例：`支持作者`）；技術相容或輔助用「支援」（範例：`支援 4K 解析度`）。

* **在地化的邊界 (Boundaries of Localization)：**
    * **判定原則：** 在地化程度應視個案認定。
    * **一般性內容（範例：語言清單）：** 應優先進行「在地化」，使用臺灣社會最通俗且具共識的稱呼。範例：`臺灣 (TW)`、`香港 (HK)`、`中國大陸` 等。
    * **特定意識形態媒介：** 若原文涉及特定宣傳或意識形態，則**不一定**需要強制在地化，以免扭曲原意或誤導讀者。
    * **敏感詞與稱呼（範例：`中國臺灣`）：**
        * 承上，若屬一般性內容，應依「在地化優先」原則修正。
        * 若有掛名或發行疑慮，應主動與作者溝通說明臺灣市場現況，目標是避免使用者產生負面觀感，確保產品被正確翻譯。

## 4. 人稱與語氣 (Tone of Voice)

* **對話情境（NPC）：**依語氣而定。
* **答謝情境（系統對玩家）：**採用「您」，範例：`感謝您的第三回遊玩！`
* **疑問情境（系統對玩家）：**採用「您」，範例：`您覺得本店特製的迎賓酒味道如何？`
* **告知情境（系統對玩家）：**採用「你」，範例：`你的錢包裡有 114,514 元`。

* **注意性別中立 (Gender Neutrality)：**避免使用帶有性別色彩的詞彙。
    * **單數 they：**採「他」或「它、牠、祂」。
    * **複數 they：**視語境而定，尤其要充當考量人物的性別意識。
    * **表示「另外的」：**一律使用「他」（這裡的他已經不是第三人稱代詞）。

## 5. 變數與日期格式 (Variables & Date Format)

* **變數處理 (Variable Handling)：** **嚴禁翻譯代碼變數**（範例：`%s`、`{count}`）。變數前後應依據「中西文間距」規範留白。

* **日期格式 (Date Format)：**由大至小排列，建議為 `YYYY/MM/DD`。範例：`2026/01/03`。

* **時間格式 (Time Format)：**採用 24 小時制。範例：`14:30`。
